# Cloudlab Project Wiki

## Overview
This is a Server-Side Rendered (SSR) web application utilizing a modern Rust stack for both the backend logic and the frontend visual layer.

- **Frontend & Reactivity**: **Leptos**
- **Backend Server**: **Axum**
- **Async Runtime**: **Tokio**
- **Client Deployment Target**: **WebAssembly (`wasm32-unknown-unknown`)**

## Project Structure
The project was originally scaffolded using the `leptos-rs/start-axum` template. 
- `src/main.rs`: Entry point for the Axum server. It builds the HTTP router, integrates it with the Leptos SSR routes, and maps out the static file directories.
- `src/lib.rs`: Connects the main App logic and handles the WebAssembly client-side hydration flow.
- `src/app.rs`: Contains the root Leptos UI `#[component]` layout. This serves as the foundation for the visual interfaces.
- `Cargo.toml`: Managed with features for `ssr` and `hydrate` allowing dual-compilation behavior between server execution and client Wasm loading.

## Development Workflow
The app requires `cargo-leptos` to be run correctly and builds two separate targets concurrently (server-side binary and client-side WASM).

```bash
# Start the live-reloading development server
cargo leptos watch

# Build for release
cargo leptos build --release
```

## Essential Guidelines for LLMs
For AI coding assistants interpreting this codebase:
- **Rust First**: The entire web logic, spanning from server routes to client-side DOM interactivity, is managed in Rust using Leptos Signals and Contexts.
- **Dual Execution**: Keep in mind that component logic generally runs twice: once on the Axum server rendering HTML, and again on the browser where it hydrates WebAssembly interactions. Code using strictly browser or backend API requirements (e.g. `window`, `fs`) must be carefully gated or run within explicit `create_effect` hooks or server blocks.
- **Architecture**: Stick to component-driven design when modifying UI, maintaining strong types inside AppState when extending the backend.

## Infrastructure & Lab Systems

### Host Onboarding & Setup
CloudLab natively registers bare-metal hosts using raw SSH connections.
- **Authentication**: When a host is configured, CloudLab manages an active SSH session pipeline (supporting both password and key/passphrase authentication). 
- **Networking (Bridging)**: For VMs to be accessible on the same network as the host, a physical bridge (typically named `br0`) must be configured on the host. CloudLab prioritizes `br0` and `br1` for virtual interfaces. If no bridge is found, it fallbacks to libvirt's default `virbr0` (NAT).
    - *Setup Hint*: Use `netplan` or `nmcli` to create `br0` and add the physical NIC (e.g. `enp...`) as a port.
- **ZFS Foundation**: Hosts provisioned into CloudLab are assumed to deploy a ZFS storage pool named `cloudlab_pool` (block) and `cloudlab_zpool` (volumes) spanning the raw `/dev/sdb` disk partitions.
- **Dynamic Capacity**: CloudLab actively calculates Total Capacity by executing parallel SSH calls to read the actual disk footprints generated by the `cloudlab_zpool`, avoiding hard-coded UI values.

### Storage & Linstor Configuration
- Storage operations (creating block devices, assigning replicas, uploading ISOs) rely on a combination of Linstor's API and backend shell orchestration. 
- **Volumes**: Linstor constructs logic-bound placement arrays (`cloudlab-isos`, etc.) dynamically mapped to `/mnt/isos/` on the hosts. 
- **Permissions**: ISO volumes must be world-readable (`755` for directories, `644` for files) to allow the `libvirt-qemu` user to access media. CloudLab automatically attempts to set these permissions during volume initialization and file uploads.
- **ISO Discovery**: The dashboard retrieves ISO artifacts via recursive `find` searches through the directories, avoiding flat-directory constraints. 
- **Status Monitoring**: CloudLab integrates directly with Linstor REST nodes via `http://<host>:3370` to query cluster health diagnostics.
- **Resource Groups**: New volumes are provisioned using the `DfltRscGrp` resource group.
- **Node Reconciliation**: When a host IP changes (e.g., after bridge setup), CloudLab automatically reconciles the Linstor node interface using `linstor node interface modify`.
- **Hostname Sync**: For reliable capability discovery (e.g., ZFS support), the host system's hostname MUST match the Linstor node name. CloudLab enforces this using `hostnamectl`.
- **Discovery Delays**: A 5-10 second delay is implemented during setup to allow the Linstor controller to discover satellite capabilities after network or property updates.
- **Idempotency**: The deployment pipeline handles "already exists" errors gracefully to allow retries.

### Virtual Machine Console (VNC)
- **Web Proxy**: CloudLab uses `websockify` to bridge the VNC TCP port (5900+) to a web-friendly port.
- **Port Range**: Websockify instances are spawned starting at port **6080** (base 6080 + display index) to avoid browser-restricted port ranges (like 6000-6007).
- **noVNC**: The host must have `novnc` installed at `/usr/share/novnc/` to serve the console UI.

### AI Inference & Fox Engine
- **Host-Native Deployment**: LLMs deploy utilizing the **Fox Inference Engine**, a custom binary dynamically fetched and compiled natively on the remote host's bare-metal layer (e.g. optimizing explicitly for CUDA paths, Pascal GPUs, Apple Silicon). This is explicitly *not* a containerized workload, guaranteeing non-abstracted hardware acceleration.
- **HuggingFace Integration**: Models are registered through dynamic queries to HuggingFace, bound directly to specific bare-metal hosts, and pulled using the Fox client synchronizers.
- **Secure ChatInterface**: Chat routines are tunneled synchronously between the Web UI and the localized Fox runtime, granting zero-latency query executions tracked via isolated session parameters.
- **Chat Persistence**: The UI maintains chat state across tab switches using CSS visibility management instead of re-mounting components.
